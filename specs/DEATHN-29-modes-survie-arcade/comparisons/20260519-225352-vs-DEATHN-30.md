## Implementation Choices Analysis

### [Spec] FR-001 — Menu labels in French ("Survie")
| Ticket | Approach |
|--------|----------|
| DEATHN-29 | `MENU_ITEMS = { "SURVIE", "ARCADE", "SIMULATION", "ZEN", "SOUND", "QUIT" }` — matches spec literal |
| DEATHN-30 | `MENU_ITEMS = { "SURVIVAL", "ARCADE", "SIMULATION", "ZEN", "SOUND", "QUIT" }` — kept English |

**Best**: DEATHN-29 — Spec explicitly names the modes in French (the title itself is "Modes Survie, Arcade…").

### [Spec] FR-012 — Simulation as a first-class mode
| Ticket | Approach |
|--------|----------|
| DEATHN-29 | Adds `.simulation` to `GameMode` enum; `startGame(.simulation, ...)`; high-score switch returns `null` for simulation, removing the `bot_tainted` runtime coupling |
| DEATHN-30 | Reuses `.survival` enum and toggles `bot_active = true` from the menu; no `.simulation` value exists. Relies on `bot_tainted` to suppress save |

**Best**: DEATHN-29 — A dedicated enum value matches the spec ("four game modes") and removes the implicit coupling between Simulation behavior and Survival mode rules.

### [Spec] FR-016/FR-017 — Heart loss/restore feedback (visual + audio)
| Ticket | Approach |
|--------|----------|
| DEATHN-29 | `drawHeartsHud()` renders 3 `<3` icons with a `heart_flash_timer` that color-flashes on loss (CRT_ERR) and restore (CRT_ACCENT); calls `playErrorSound()` and `playPowerUpSound(.shield)` |
| DEATHN-30 | `drawArcadeLivesHud()` prints `LIVES: <3 <3 <3` in CRT_ERR; no flash, no audio cue |

**Best**: DEATHN-29 — FR-016 explicitly requires audio AND visual feedback.

### Decision Point: Wave-mode gating abstraction
| Ticket | Approach | Constitution-style | Trade-offs |
|--------|----------|--------------------|-----------|
| DEATHN-29 | `fn isWaveMode(mode) bool` helper switches on enum exhaustively | Better — exhaustive switch is forward-compatible | One extra function |
| DEATHN-30 | Inline `mode != .zen` checks repeated 4× | Works but brittle when a new mode is added | Smaller diff, but easy to forget a site |

**Best**: DEATHN-29 — Exhaustive enum switch survives future mode additions.

### Decision Point: Test coverage
| Ticket | Tests added |
|--------|------------|
| DEATHN-29 | ~15: MAX_HEARTS, hearts start per mode, restore cap, power-up isolation, multi-loss, shield priority, simulation never saves, BOT-string absence, simulation activates bot, menu labels (six), per-mode score independence |
| DEATHN-30 | ~5: lives start constants, reset preserves, regen cap, decrement-to-zero game-over, menu labels |

**Best**: DEATHN-29 — Broader coverage of the new behavior, including FR-002 ("no `BOT` literal in MENU_ITEMS").

## Ranking & Recommendation

### 1. **DEATHN-29** — Best implementation (95%)
**Why it wins**:
- Full spec fulfillment (18/18) including French menu literals and audio+visual heart feedback
- Dedicated `.simulation` enum value matches the spec's "four game modes" model
- Forward-compatible `isWaveMode()` abstraction over inline `mode != .zen` checks
- ~3× more tests for the new behavior, including a regression test against the literal `"BOT"`
- Stats overlay shows the mode label (extra UX, surfaces which mode produced the score)

### 2. **DEATHN-30** — (78%)
**Why it lost**:
- Menu still says `SURVIVAL` (FR-001 literal violation)
- No `.simulation` enum — Simulation is implemented as `.survival` + bot flag (FR-012 partial)
- Heart loss/restore has neither audio nor flash (FR-016/17 partial)
- Smaller test surface — only 5 new tests vs ~15
