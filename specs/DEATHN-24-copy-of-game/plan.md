# Implementation Plan: Game-Over Stats Screen and High Score Persistence

**Branch**: `DEATHN-24-copy-of-game` | **Date**: 2026-05-17 | **Spec**: `specs/DEATHN-24-copy-of-game/spec.md`
**Input**: Feature specification from `specs/DEATHN-24-copy-of-game/spec.md`

## Summary

Replace the minimal game-over display with a full-screen statistics screen showing wave, score, best score, average WPM, accuracy, and kills. Add a 1-second death transition with red-tinted zombie highlighting. Persist the high score to `highscore.dat` (native binary) or `localStorage` (web/emscripten) so it survives across sessions. All changes are in `src/main.zig` with minor additions to `src/raylib.zig` for emscripten JS interop.

**Language/Version**: Zig (0.16-dev, toolchain-installed)
**Primary Dependencies**: raylib (pinned commit, static linked), Emscripten SDK 3.1.64 (web build)
**Storage**: `highscore.dat` binary file (native), `localStorage` key `death-note.highscore` (web)
**Testing**: `zig build test` â€” Zig built-in test runner

**Project Type**: Single-module game
**Performance Goals**: 60 FPS maintained; file I/O only at startup and game-over (not per-frame)
**Constraints**: 800Ă—450 fixed window; no network; no dynamic allocation for persistence
**Scale/Scope**: ~1300 LOC in main.zig; single-player local game

## Implementation Phases

Add new module-level state variables and the `total_kills` counter.

   - `is_dying: bool = false`
   - `dying_timer: f32 = 0.0`
   - `dying_zombie_index: ?usize = null`

3. Add named constants for stats screen layout:

### Phase 2: Game-Over Transition (1-Second Death Animation)

Implement the DYING state between PLAYING and GAME_OVER.

**Changes to `src/main.zig`**:
1. In `updateZombies`: when `zomb.y >= screen_height`, instead of `is_game_over = true`, set `is_dying = true`, `dying_timer = DYING_DURATION`, `dying_zombie_index = i` (the slot index). Return immediately.
2. In `updateBoss`: same pattern â€” when `b.y >= screen_height`, set `is_dying = true` and track it.
3. In `frame()`: add `!is_dying` to the `!is_game_over and !is_transitioning` gate (line 173) so no updates happen during the dying state.
4. Add dying state countdown logic in `frame()`: if `is_dying`, decrement `dying_timer` by frame time. When `dying_timer <= 0`: set `is_game_over = true`, `is_dying = false`, check and save high score.
5. In `drawZombies`: when `is_dying` and the zombie index matches `dying_zombie_index`, draw that zombie with `raylib.RED` tint instead of `raylib.WHITE`.
6. During `is_dying`, still draw all zombies and boss (frozen in place, with the culprit tinted red).

**Tests**: Test dying state transition logic (is_dying set correctly, timer countdown).

### Phase 4: High Score Persistence â€” Native

Implement file-based persistence for native builds.

**Changes to `src/main.zig`**:
1. Add `loadHighScore() !HighScoreRecord` function:
   - Open `HIGHSCORE_FILENAME` for reading
   - Read exactly `@sizeOf(HighScoreRecord)` bytes
   - If file doesn't exist or size mismatches â†’ return zero-initialized record
   - Use `@as(*const HighScoreRecord, @ptrCast(@alignCast(bytes.ptr))).*` to reinterpret
2. Add `saveHighScore(record: HighScoreRecord) !void` function:
   - Create/overwrite `HIGHSCORE_FILENAME`
   - Write `@as([*]const u8, @ptrCast(&record))[0..@sizeOf(HighScoreRecord)]`
3. In `main()`, after window init: `best_score = loadHighScore() catch HighScoreRecord{ .score = 0, .wave = 0, .wpm = 0, .accuracy = 0 }`
4. At game-over (when dying timer expires): if `score > best_score.score`, build new record and call `saveHighScore`, update `best_score` in memory
5. Gate all of the above behind `comptime @import("builtin").target.os.tag != .emscripten`

**Tests**: Test `HighScoreRecord` struct size matches expected 17 bytes. Test round-trip serialization logic (write bytes, read back, compare fields).

### Phase 5: High Score Persistence â€” Web (Emscripten)

Implement localStorage-based persistence for the web build.

**Changes to `src/raylib.zig`**: Verify that `emscripten_run_script` and `emscripten_run_script_int` are accessible through the existing `@cImport`. They should be â€” `emscripten/emscripten.h` declares them.

**Changes to `src/main.zig`**:
1. Add `loadHighScoreWeb() HighScoreRecord` function:
   - Call `raylib.emscripten_run_script_int("...")` with JS that reads `localStorage.getItem('death-note.highscore')`, parses JSON, returns the score field
   - Use multiple calls or a single script that encodes all 4 fields into a packed integer (score fits u64, so may need multiple calls or a string-based approach)
   - Alternative: use `emscripten_run_script` to set a global JS variable, then read fields individually
   - On any failure (null, parse error) â†’ return zero record
2. Add `saveHighScoreWeb(record: HighScoreRecord) void` function:
   - Build a JS string like `localStorage.setItem('death-note.highscore', JSON.stringify({score:N,wave:N,wpm:N,accuracy:N}))`
   - Call `raylib.emscripten_run_script(js_string)`
3. Gate behind `comptime @import("builtin").target.os.tag == .emscripten`
4. At startup and game-over, the same logic as Phase 4 but dispatching to web variants

**Tests**: Limited â€” web persistence requires a browser. Add a compile-time test that the emscripten branch compiles (i.e., no syntax errors in the comptime-gated code). Manual test note in PR.

### Phase 6: Restart State Reset

Ensure the restart handler properly resets all new state while preserving `best_score`.

**Changes to `src/main.zig`**:

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| File writes (`highscore.dat`) | Spec explicitly requires persistence (FR-008); ARD-1 documents human authorization | In-memory only loses high scores on exit â€” defeats the feature purpose |
| `emscripten_run_script` for JS interop | Spec requires web persistence via localStorage (FR-009) | No simpler Zigâ†’JS bridge exists in the emscripten toolchain without a custom JS library |
| New game state (`is_dying`) | Spec requires 1-second transition with frozen gameplay (FR-001, FR-002) | Reusing `is_game_over` with a timer conflates two visually distinct states and complicates the draw logic |
